# 实验规则

## 状态

- status: active
- source: 用户确认和当前项目适配层规划
- updated: 2026-05-15

## 结论

实验是为了验证不确定技术路线、生成质量、可玩性、资产链路或自动验证能力。实验不能直接替代正式开发任务。

## 启动条件

满足任一条件时先做实验：

- 技术路线不确定。
- 生成结果质量不可预估。
- 需要比较多个方案。
- 会产生大量临时产物。
- 需要验证 Godot、资产、模型或自动测试链路。

## 命名规则

实验 ID 使用：

```text
EXP-YYYYMMDD-三位序号-短名称
```

示例：

```text
EXP-20260515-001-gamespec-minimal
```

## 单实验目录

每个实验目录至少包含：

```text
plan.md
manifest.md
result.md
inputs/
workspace/
artifacts/
logs/
captures/
```

## 晋升规则

- `result.md` 只代表单次实验结果。
- 只有经过复核的结论才能写入 `findings.md`。
- 只有稳定、可复现、与产品目标相关的结论才能写入正式 wiki。
- 失败实验必须保留失败原因，不能删除后伪装成未发生。

## 验证

实验完成时必须说明：

- 实验目标是否达成。
- 使用的输入和配置。
- 执行命令或手动步骤。
- 输出产物位置。
- 可复现性。
- 是否建议进入开发任务。
