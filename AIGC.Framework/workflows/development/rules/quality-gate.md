# 质量门控规则

## 结论

开发交付前必须做最小质量门控，未通过时先补齐缺口，再交付给用户。

## 触发条件

- 输出开发计划、技术方案、文档或代码交付物前。
- 多个页面、模块或执行者共同产出同一结果时。
- 本轮改动影响验证方式、项目 wiki、运行记录或通用规则时。
- 交付 OneHarness 自身改动前。

## 最小门控

| 门控项 | 通过条件 |
| --- | --- |
| 输入完整 | 目标、范围、非目标、项目 wiki 策划页面入口、成功标准和验证方式已明确。 |
| 任务页完整 | 项目 wiki 主任务页、子任务页、分配角色、写入范围和接口契约已记录。 |
| 会话隔离 | 每个子任务由全新 SubAgent 或全新会话执行；无法满足时已有替代隔离说明。 |
| 范围收敛 | 修改文件都能追溯到本轮目标，没有顺手改无关内容。 |
| 验证可执行 | 每个成功标准都有命令、测试、手动步骤或无法验证说明。 |
| 边界干净 | 项目事实没有写入通用 AIGC，临时过程没有写入 wiki，候选知识已有处理理由。 |
| 结果一致 | 交付说明、项目 wiki 任务页、修改文件、验证记录和 wiki/运行记录判断一致。 |
| 自检通过 | 命中 `self-check.md` 的命令已运行；交付前 `python tools\oneharness.py gate` 已通过。 |

## 不通过处理

- 缺输入：回到 `intake.md`。
- 缺任务拆解：回到 `task-breakdown.md`。
- 缺角色路由：回到 `game-role-routing.md`。
- 缺验证：回到 `verification.md`。
- 缺自检：回到 `self-check.md`。
- 写入边界错误：回到 `wiki-sync.md` 和项目适配规则。
- 多执行者冲突：回到 `lifecycle.md`，明确文件归属。
- 候选知识未审核：回到 `wiki-sync.md`。

## 禁止

- 禁止把门控变成全仓扫描。
- 禁止为了门控读取无关历史资料。
- 禁止未通过门控仍标记完成。
