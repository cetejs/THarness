# 入口规则

## 目标

判断本轮游戏开发工作处于哪个阶段，明确写入边界、成功标准和不能做的事。

## 必须确认

- 用户当前要的是讨论、文档、开发、修复、验证、复盘还是流程沉淀。
- 是否允许写文件；如果允许，写入目标是通用 AIGC 还是目标项目 wiki。
- 目标引擎、运行语言、平台、编辑器或工具链是否已确认。
- 当前是否已有项目 wiki 任务页、接口契约、子任务和验证方式。
- 本轮的成功标准是否可验证。

## 阶段判断

| 信号 | 阶段 |
| --- | --- |
| 只有玩法想法或参考游戏 | 策划和循环阶段 |
| 已要求每个界面、按钮和跳转 | 界面合同阶段 |
| 已讨论素材、风格、效果图或占位资源 | 视觉和素材阶段 |
| 已要求完整游戏、配置、数据表或内容扩展 | 数据蓝图阶段 |
| 用户明确说可以开发 | 开发门控阶段 |
| 已出现编译、运行、截图或布局问题 | 排查阶段 |
| 用户要求总结经验或沉淀方法 | 复盘沉淀阶段 |

## 输出要求

- 用一句话说明当前阶段。
- 列出本轮允许写入和禁止写入的范围。
- 如果进入开发，必须引用目标项目任务、契约和验证方式。
- 如果进入通用沉淀，必须去掉具体项目事实，只保留跨项目方法。

## 禁止

- 不把用户的“准备补充”理解成“开始开发”。
- 不把一次截图反馈直接写成稳定项目事实，必须先经过验证或记录为问题。
- 不在通用流程中保存具体项目的数值、素材路径、角色名或运行日志。
