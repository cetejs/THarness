# 游戏开发生命周期工作流

## 目标

把游戏项目从想法、策划、视觉、数据、开发、运行时排查、验证到复盘沉淀为一条可重复执行的闭环流程。

本工作流只保存跨项目通用方法，不保存具体项目的角色、卡牌、数值、素材路径、引擎路径、运行日志或项目事实。

## 激活条件

用户明确表达以下任一意图时激活：

- 需要把一次游戏开发过程总结成可复用流程。
- 需要从策划推进到可玩的垂直切片，并持续迭代。
- 需要建立游戏项目的开发节奏、质量门控、验证方式和复盘机制。
- 游戏开发中反复出现 UI、资源、运行时或构建问题，需要建立不靠猜测的排查流程。
- 设计稿、效果图和实机截图差异明显，需要复盘并补充视觉一致性门控。
- 需要判断一个游戏项目是否可以进入实现、继续扩展内容或进入下一轮迭代。

## 与其他工作流的关系

| 工作流 | 关系 |
| --- | --- |
| `planning-discussion` | 需求、目标体验、参考游戏、范围和成功标准尚不明确时使用。 |
| `game-iteration` | 用于把游戏想法拆成最小可玩循环、界面拆分、数据蓝图和开发门控输入包。 |
| `development` | 用于执行已确认的开发、修复、重构、验证和交付。 |
| `project-wiki-maintenance` | 用于把目标项目事实、任务、契约、运行记录和追溯记录写入目标项目 wiki。 |
| `capability-evolution` | 当本工作流需要升级为通用 AIGC 能力版本时使用。 |

## 默认读取

1. 本文件。
2. `rules/INDEX.md`。
3. 命中阶段规则。
4. 需要模板时读取 `templates/INDEX.md` 和命中的模板。
5. 需要目标项目事实时，通过 `../../projects/INDEX.md` 路由到目标项目适配层。
6. 需要实际开发时，交接 `../development/WORKFLOW.md`。

## 阶段

| 阶段 | 输出 | 验证 |
| --- | --- | --- |
| 入口和边界 | 生命周期任务摘要 | 目标、引擎、运行语言、写入边界、禁止范围和成功标准明确。 |
| 最小可玩循环 | 玩家从开始到结算的闭环 | 每个节点都有玩家操作、反馈、失败路径和下一步跳转。 |
| 玩家界面拆分 | 玩家可见界面清单和 UI 合同 | 每个界面有布局、按钮、数据来源、跳转和状态反馈。 |
| 视觉和素材准备 | 风格基准、效果图、素材策略 | 参考图、效果图、占位素材和可用运行时素材边界清楚。 |
| 素材管线合同 | 运行时素材来源、生成、导入和校验合同 | 源资源、运行时资源、生成脚本、缓存刷新和缺失策略可复查。 |
| 设计实现一致性门控 | 目标截图、运行时界面合同和差异验收标准 | 效果图已拆成分辨率、布局比例、层级、素材、字体、交互状态和允许偏差。 |
| 数据蓝图 | 数据模块、字段、引用和配置项 | 玩法对象、UI 参数、素材引用、存档和验证器能闭合。 |
| 开发前硬门禁 | 开发前检查清单 | 工具链、目标语言、逻辑分辨率、缩放策略、界面合同、素材合同和验证命令齐备。 |
| 开发门控 | 主任务、子任务、契约和验证计划 | 没有任务页、契约和验证方式时不得进入实现。 |
| 垂直切片实现 | 最小可玩版本 | 先跑通完整闭环，再扩内容；代码只写入允许范围。 |
| 视觉校准和日志排查 | 截图差异、布局日志和修复记录 | 反复问题必须先加日志或观测点，再基于证据修复。 |
| 验证和交付 | 构建、数据、smoke、截图或手动验证记录 | 自动验证和必要的人工验证都有结果；无法验证时说明风险。 |
| 项目侧执行同步 | 项目适配层同步检查 | 当前工作流、状态页、构建运行入口、任务状态、运行记录和追溯索引一致。 |
| 复盘沉淀 | 项目记录和通用候选知识 | 项目事实写项目 wiki，通用方法写 THarness，未确认内容写开放问题。 |

## 核心原则

- 先定义玩家能玩到的闭环，再写实现。
- 先确认运行时和工具链支持目标语言，再迁移或新增代码。
- UI 先有目标效果和界面合同，再进入工程落地。
- 效果图不能只作为风格参考；必须转成可执行、可截图对比的运行时界面合同。
- 分辨率或缩放策略变更后，必须重新确认目标设计稿或布局比例合同。
- 开发前必须把工具链、目标语言、默认分辨率、缩放策略、界面合同、素材管线和验证命令合并成一份可核对清单。
- 数据、配置和素材引用必须能被验证器检查。
- UI 核心对象必须有明确玩家语义；看起来像占位的空框、空槽或无文本控件不能进入验收版本。
- 截图验证必须和真实输入验证配套，不能用截图代替可点击、可拖拽或可跳转的验收。
- 运行时资源替换必须考虑引擎导入缓存，生成或覆盖资源后要刷新导入或使用明确的缓存替换策略。
- 重复出现的问题不继续猜；先增加日志、截图、运行参数或最小复现。
- 通用工作流升级后必须回查目标项目适配层，不能让当前工作流、验证入口、任务状态或追溯索引停留在旧版本。
- 默认先做最小可玩垂直切片，不用“完整游戏”作为第一轮内容规模。
- 每次正式新增任务、契约、文件、稳定结论或验证记录都要可追溯。

## 完成标准

必须满足：

- 本轮属于哪个阶段、输出什么、写到哪里已经明确。
- 目标项目事实没有写入通用 AIGC。
- 开发前存在目标项目 wiki 任务、子任务、接口契约和验证方式。
- 开发前硬门禁清单已完成，且未完成项没有被默认视为可开发。
- 运行时、引擎版本、语言版本和入口命令已经验证。
- 每个玩家可见界面都有布局、按钮、跳转、状态和错误反馈。
- 关键界面都有目标设计图、运行时截图、差异记录和修复计划。
- 每个动态尺寸、数值、路径和开关都来自配置或数据，不靠散落硬编码。
- 构建检查、数据检查、最小闭环 smoke 和必要的人工截图验证完成。
- 项目侧执行同步检查已完成，状态页、构建运行入口、任务索引、运行记录索引和追溯索引没有互相矛盾；未同步项已记录为风险或下一步。
- 复盘时明确哪些是项目事实，哪些是跨项目可复用方法。

## 禁止

- 禁止把具体项目内容写入本通用工作流。
- 禁止没有任务页和接口契约就进入开发。
- 禁止用全局滚动框、强行缩放或遮罩掩盖布局错误，除非界面体验本身需要滚动。
- 禁止把参考图误当成可直接使用的运行时素材。
- 禁止把“无编译错误”或“能进入界面”当成视觉落地完成。
- 禁止在未验证工具链前承诺某种语言、引擎版本或导出链路可用。
- 禁止反复修改同一类问题却不增加日志、截图或复现入口。
