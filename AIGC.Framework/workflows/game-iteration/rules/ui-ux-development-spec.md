# UI/UX 可开发规格规则

## 目标

把玩法策划案补成 UI 开发可以执行、测试和验收的规格，避免只靠风格描述、截图和主观判断反复修界面。

## 触发条件

出现以下情况时必须补本规则：

- 用户认为 UI 开发耗时过长，怀疑策划输入不足。
- 策划案只有玩法系统和界面名称，没有工程可执行的布局和验收标准。
- 已有参考图或效果图，但没有拆成分辨率、区域比例、组件规格和交互合同。
- UI 需要进入实现、重做或验收。

## 必须补充

| 项 | 要求 |
| --- | --- |
| 玩家界面清单 | 每个玩家可见界面的目标、来源、去向和失败路径。 |
| 目标视口 | 默认逻辑分辨率、验收视口、窗口缩放策略和不允许裁切的区域。 |
| 界面布局合同 | 每个界面的主要区域、占屏比例、层级、留白、滚动策略和遮挡关系。 |
| 交互合同 | 每个按钮、卡牌、列表项和节点的可用条件、反馈、跳转、禁用态和失败反馈。 |
| hitbox 合同 | 关键控件的最小点击尺寸、目标视口内约束、缩放后规则和真实输入验收。 |
| 组件规范 | 卡牌、道具、技能、角色、敌人、地图节点、弹窗、按钮等复用组件的结构和状态。 |
| 视觉拆解 | 参考图、效果图和运行时素材的边界；构图比例、字体层级、材质和允许偏差。 |
| 素材边界 | 区分参考图、效果图、占位素材、运行时素材和正式验收素材。 |
| 验收标准 | 截图验收、真实输入验收、缩放验收、回归验收和不可接受差异。 |

## 策划案不足的判定

任一情况存在时，策划案不能直接交给 UI 开发：

- 只写“界面显示某系统”，没有区域布局、按钮和跳转。
- 只写“风格参考某图”，没有设计稿到运行时拆解。
- 只写“适配窗口”，没有目标分辨率和缩放策略。
- 只写“按钮可点击”，没有禁用态、反馈、失败路径和真实输入验收。
- 只写“使用正式素材”，没有素材来源、格式、运行时引用和缺失策略。
- 只写“好看”或“符合风格”，没有截图差异和允许偏差标准。

## 输出要求

输出 UI/UX 可开发规格时，至少包含：

1. 目标视口和缩放策略。
2. 玩家界面总表。
3. 每个界面的布局合同。
4. 核心组件规范。
5. 交互和 hitbox 合同。
6. 素材边界和运行时素材策略。
7. 验收截图、真实输入和回归标准。
8. 不属于本轮 UI 开发的内容。

## 禁止

- 禁止把玩法策划案直接当 UI 工程合同。
- 禁止用单张效果图替代布局、交互、hitbox 和验收标准。
- 禁止只写视觉风格，不写玩家操作和状态反馈。
- 禁止只用截图验收可交互界面。
