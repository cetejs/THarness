# 游戏数据蓝图规则

## 目标

在开发前定义完整游戏需要的数据能力，避免运行时硬编码玩法内容。

数据蓝图不是 schema、不是配置实现、不是代码。用户未明确授权开发前，只能输出文字和表格。

## 必须覆盖的数据模块

| 模块 | 说明 |
| --- | --- |
| 角色或玩家原型 | 初始资源、初始构筑、可用内容池。 |
| 核心玩法对象 | 卡牌、技能、单位、道具或动作等。 |
| 构筑扩展 | 辅助、装备、词缀、天赋、升级或改造。 |
| 状态和效果 | 状态、持续时间、叠加规则、结算时机。 |
| 敌人或挑战 | 属性、行为、意图、难度层级。 |
| 遭遇或关卡 | 挑战组合、地图节点、章节配置。 |
| 事件和商店 | 非核心战斗选择、代价和收益。 |
| 奖励池 | 资源、道具、构筑项和跳过补偿。 |
| UI 流程 | 界面、按钮动作、跳转、数据来源。 |
| 素材引用 | 图标、卡框、角色、敌人、背景和占位策略。 |
| 存档 | 一局游戏或长期进度需要保存的状态。 |

## 字段定义要求

每个模块至少说明：

- 唯一 ID。
- 显示名称。
- 类型或分类。
- 数值字段。
- 引用字段。
- 效果字段。
- 解锁或出现条件。
- 素材引用。
- 验证方式。

## 引用关系要求

必须输出引用关系图或引用表，例如：

```text
角色 -> 初始内容 -> 核心玩法对象
核心玩法对象 -> 标签 -> 构筑扩展
构筑扩展 -> 效果 -> 状态或资源
遭遇 -> 挑战对象 -> 奖励池
地图节点 -> 遭遇/事件/商店/Boss
UI -> 数据模块 -> 素材引用
存档 -> 当前局内状态
```

所有引用必须能追溯来源。未确认引用只能写入待确认问题，不能写成事实。

## 内容规模分层

数据蓝图必须区分：

- 原型验证内容量。
- 第一阶段完整内容量。
- 多章节体验内容量。
- 完整游戏目标内容量。

## 数值基准

至少给出第一阶段估算基准：

- 玩家资源。
- 核心行动成本。
- 普通挑战强度。
- 精英或高难挑战强度。
- Boss 或章节终点强度。
- 奖励收益范围。

数值基准只用于开发前估算，不能替代平衡测试。

## 验收标准

数据蓝图完成必须满足：

1. 核心模块都有字段定义。
2. 一局从开始到结算的数据链完整。
3. 所有 ID 都能追溯来源。
4. 所有 UI 都有数据来源。
5. 所有玩法行为都能落到效果字段。
6. 所有奖励都能追溯到奖励池。
7. 素材引用有占位策略。
8. 未确认内容没有写成已确认事实。

## 禁止

- 禁止在蓝图阶段直接创建 schema、配置文件或验证脚本，除非用户明确授权开发。
- 禁止把数值、敌人、奖励或素材路径硬编码为运行时事实。
- 禁止把项目专用数据写入 THarness。
