# 游戏开发门控规则

## 目标

判断游戏项目是否可以从准备阶段进入开发执行。

## 可以进入开发的最低条件

必须同时满足：

- 已确认最小可玩循环。
- 已确认界面清单、按钮动作和跳转。
- 已确认 UI/UX 可开发规格，包含目标视口、缩放策略、界面布局合同、组件规范、hitbox、素材边界和验收标准。
- 已确认数据蓝图。
- 已确认第一阶段范围。
- 目标项目 wiki 中存在主开发任务页。
- 目标项目 wiki 中存在接口契约。
- 目标项目 wiki 中存在子任务页。
- 每个子任务有允许读取入口、允许修改范围、禁止范围和验证方式。
- 有验证计划。
- 用户明确确认进入开发。

## 推荐第一阶段范围

第一阶段应验证“完整结构能力 + 最小内容链路”，不应直接填满完整游戏内容量。

推荐表达：

```text
第一阶段 = 1 个核心玩家原型 + 1 个完整章节或关卡组 + 1 个终点挑战 + 主菜单到结算的完整链路。
```

具体数量由目标项目决定，不能写入 THarness。

## 进入开发后的优先级

推荐顺序：

1. 数据 schema 或等价数据结构。
2. 第一阶段样例数据。
3. 数据引用验证。
4. 引擎最小运行壳。
5. UI 和素材引用。
6. 核心玩法运行闭环。
7. 结算和验收。

## 不能进入开发的情况

任一情况存在时不得进入开发：

- 没有主任务页。
- 没有接口契约。
- 没有子任务拆分。
- 没有验证方式。
- 没有 UI/UX 可开发规格，却要求进入 UI 或完整游戏实现。
- 用户只要求判断、整理、计划或落文档。
- 当前问题仍是产品方向或范围选择。

## 输出格式

判断是否可开发时，输出：

- 当前判断：可以开发 / 不可以开发 / 只可进入准备阶段。
- 已满足条件。
- 未满足条件。
- 推荐下一步。
- 禁止范围。

## 成功标准

1. 没有把准备阶段误判成开发阶段。
2. 没有跳过目标项目 wiki 的任务、契约和子任务门控。
3. 没有把玩法策划案误当成 UI 工程规格。
4. 进入 development 工作流前，输入包足够执行。
