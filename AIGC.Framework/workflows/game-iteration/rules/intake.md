# 游戏迭代输入和边界规则

## 必须确认

开始游戏迭代前必须确认：

- 当前目标是讨论、落文档、生成素材、补开发门控，还是进入开发。
- 用户是否允许写入文件。
- 写入位置是目标项目 wiki、目标项目实验目录、运行记录，还是通用 THarness。
- 具体项目事实和通用流程是否已分离。
- 是否存在参考游戏、竞品、截图或素材。
- 参考内容只抽取结构，不复刻具体命名、数值、素材和世界观。

## 写入边界

| 内容 | 写入位置 |
| --- | --- |
| 具体项目事实、策划、任务、契约、验证记录 | 目标项目 wiki 或目标项目适配层。 |
| 临时讨论过程和失败尝试 | 目标项目运行记录。 |
| 实验计划、实验产物、试制素材 | 目标项目实验目录。 |
| 跨项目可复用流程、模板、规则 | THarness `AIGC/`。 |

## 开发授权

用户只说“准备”“判断”“计划”“补充文档”时，不得进入代码实现。

只有用户明确表达以下含义，才允许进入开发执行：

- 开始开发。
- 实现第一阶段。
- 创建 schema、配置、工程或脚本。
- 按已有任务页进入开发。

## 输出要求

输入阶段输出必须包含：

- 当前阶段判断。
- 已确认内容。
- 未确认内容。
- 本轮允许写入范围。
- 本轮禁止范围。
- 下一步建议。

## 成功标准

1. 用户意图没有被默默升级为开发。
2. 写入位置符合项目事实和通用流程分离规则。
3. 参考来源的使用边界明确。
