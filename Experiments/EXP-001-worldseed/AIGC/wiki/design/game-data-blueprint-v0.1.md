# 完整游戏数据设计蓝图 v0.1

## 状态

- status: draft
- source: 用户确认、策划讨论运行记录、最小可玩循环方案
- updated: 2026-05-16

## 结论

WorldSeed 第一款游戏在进入开发前，需要先完成完整游戏数据蓝图。蓝图目标不是立刻填满所有内容，而是先确认完整游戏所需的数据模块、字段、引用关系、内容规模、数值基准和验收标准。

本页是开发前准备文档，不是代码实现、不是 JSON schema、不是 Godot 工程任务。

## 内容

### 设计目标

本数据蓝图服务于“爬塔式 Roguelike 卡牌 + ARPG 深构筑”的完整游戏方向。

数据层必须支持：

- 从主菜单开始一局游戏。
- 选择角色并生成地图。
- 进入战斗并使用卡牌。
- 通过奖励、辅助插槽、装备和天赋形成构筑。
- 完成事件、商店、锻造和 Boss 战。
- 在失败或通关后进入结算。
- 后续扩展为多章节、多角色、多构筑、多 Boss 和后期地图。

### 系统引用关系

```text
角色 -> 初始牌组 -> 卡牌
角色 -> 初始装备 -> 装备
角色 -> 起始天赋 -> 天赋
卡牌 -> 标签
辅助卡 -> compatible_tags -> 技能卡标签
装备 -> 词缀 -> 效果
天赋 -> 前置天赋 -> 效果
敌人 -> 意图 -> 效果/状态
遭遇 -> 敌人 -> 奖励池
地图节点 -> 遭遇/事件/商店/Boss
章节 -> 地图配置/敌人池/事件池/商店池/Boss
奖励池 -> 卡牌/装备/金币/天赋点
UI -> 对应数据模块 -> 素材引用
存档 -> 当前局内状态
```

禁止出现没有来源的引用。所有 ID 都必须能在同一份数据蓝图或后续正式数据表中找到定义。

### 内容规模规划

| 阶段 | 目标 | 内容规模 |
| --- | --- | --- |
| 原型验证 | 验证数据结构和最小循环。 | 1 个角色、10 到 20 张牌、3 个普通敌人、1 个 Boss、1 个章节。 |
| 第一章完整 | 验证三角色和第一章体验。 | 3 个角色、60 到 90 张牌、8 到 12 个敌人、2 个精英、1 个 Boss、6 到 10 个事件。 |
| 三章体验 | 验证完整通关体验。 | 3 到 5 个角色、150 到 220 张牌、30 个敌人、6 个精英、3 个 Boss、20 到 30 个事件。 |
| 完整游戏 | 支持长期构筑和后期扩展。 | 6 到 8 个角色、多章节、多 Boss、后期地图、扩展装备和天赋池。 |

### 存档数据范围

一局游戏至少需要保存：

- 当前角色 ID。
- 当前生命、最大生命、金币。
- 当前章节和地图节点。
- 当前牌组、抽牌堆、弃牌堆和消耗区。
- 当前装备。
- 已解锁天赋。
- 技能卡和辅助卡的插槽关系。
- 当前地图生成结果。
- 已完成节点。
- 本局统计。

### 素材引用策略

第一阶段允许使用试制占位素材，但数据中必须保留素材引用字段。

素材引用不能嵌死在界面逻辑里。后续 Godot 实现应从数据读取：

- 卡框。
- 图标。
- 地图节点。
- 角色头像和战斗立绘。
- 敌人头像和战斗立绘。
- UI 面板和按钮。
- 状态效果图标。

### 数据验收标准

数据准备完成必须满足：

1. 每个核心模块都有字段定义。
2. 一局从角色选择到 Boss 结算的数据链完整。
3. 所有 ID 都能追溯来源。
4. 所有 UI 都有明确数据来源。
5. 所有战斗行为都能落到效果字段。
6. 所有奖励都能追溯到奖励池。
7. 辅助卡和技能卡的兼容规则可通过标签判断。
8. 装备、天赋、词缀都不直接写死到界面或战斗逻辑。
9. 素材引用有占位策略。
10. 未确认内容没有写成已确认事实。
