# RUN-20260516-007-combat-ui-design-parity-retro

## 状态

- status: completed
- source: 用户要求复盘斗法界面与设计稿差异并同步更新工作流
- updated: 2026-05-16

## 目标

复盘斗法界面实机效果与已确认视觉稿差异较大的原因，并把可复用经验沉淀到 THarness 通用游戏开发生命周期工作流。

## 证据

| 项目 | 路径或来源 | 结论 |
| --- | --- | --- |
| 原斗法设计图 | `WorldSeed.AIGC/wiki/design/images/ui-screen-mockups-v0.3/ui-004-combat.png` | 目标是中国风手绘卡牌界面，战场、手牌、记录区和装饰层需要形成完整构图。 |
| 实机截图 | 用户提供的运行截图 | 顶部留白过大、卡牌偏小、装饰层级弱、底部背景穿过操作区，整体不像目标设计稿。 |
| 布局日志 | 用户提供的 `[WorldSeed/UI]` 日志 | 旧节点累积问题已能通过日志定位，但日志不能判断视觉是否贴近设计稿。 |
| 对比产物 | `WorldSeed.Experiments/ui-comparison-20260516/combat-design-vs-current.png` | 差异集中在构图比例、区域层级、卡牌尺寸和背景装饰关系。 |

## 差异原因

| 原因 | 说明 | 后果 |
| --- | --- | --- |
| 缺少设计稿到运行时合同 | 效果图确认后，没有先拆成分辨率、区域比例、控件尺寸、层级、允许偏差和截图验收标准。 | 实现只能按组件可用性推进，无法判断整体是否达标。 |
| 把效果图偏向风格参考使用 | 运行时优先套用了通用面板、通用卡牌和旧布局，而不是按斗法设计稿重建构图。 | 卡框局部变像了，但页面整体仍不像目标稿。 |
| 1080p 变更没有先重建目标稿 | 默认分辨率切到 `1920x1080` 后，主要做了配置放大和自适应修正，没有先确认 1080p 目标设计。 | 旧 720p 构图比例被放大后产生顶部空白和区域失衡。 |
| 验证偏向功能和构建 | 已有验证覆盖构建、数据、smoke 和日志，但没有把“设计图 vs 实机截图”作为必过门控。 | 能运行不代表视觉完成，偏差直到用户截图才暴露。 |
| 修复节奏偏局部症状 | 先后处理滚动框、窗口缩放、卡牌尺寸、清屏累积等具体问题，但没有在每轮修复后回到目标稿整体对齐。 | 局部 bug 变少，但整体视觉体验仍偏离。 |

## 结论

这次差异不是单纯的默认分辨率错误，也不是单个控件 bug。主要问题是开发流程缺少“设计实现一致性门控”：视觉稿确认后没有转成运行时界面合同，导致工程实现可以通过构建和基础交互验证，却不能保证与设计目标一致。

## 后续建议

- 先补一份 `1920x1080` 斗法界面合同，明确顶部资源区、战场区、手牌区、斗法记录区和背景装饰的比例。
- 再基于合同修正 C# UI，而不是继续直接凭截图微调。
- 每次 UI 修复后保留实机截图，和目标设计图一起进入验证记录。
