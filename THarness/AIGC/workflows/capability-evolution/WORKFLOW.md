# AIGC 能力演化工作流

## 目标

从外部 harness、旧项目、工具链、文档或历史经验中提取可复用能力，并以最少 token 成本更新当前 AIGC 的能力索引、通用架构知识、规则、模板或版本记录。

## 激活条件

用户明确表达以下任一意图时激活：

- 分析、对比或吸收其他 harness 工程。
- 从旧项目中提取可复用 AIGC 能力。
- 判断某个外部结构是否值得合入当前 AIGC。
- 更新 AIGC 能力索引、版本信息或变更记录。
- 演进通用工作流、规则、模板或 wiki 结构。

## 默认读取

1. 本文件。
2. `../session-visible-state.md`。
3. `rules/INDEX.md`。
4. `../../capabilities/INDEX.md`。
5. `../../wiki/INDEX.md`。
6. 按 `read_when` 命中的规则继续读取。

## 阶段

| 阶段 | 输出 | 验证 |
| --- | --- | --- |
| 来源登记 | 来源摘要 | 来源、目标、读取边界明确。 |
| 最小读取 | 读取清单 | 只读取入口、索引和命中正文。 |
| 能力对比 | 差异清单 | 已列出已有能力、缺失能力、可补全能力和不采纳内容。 |
| 候选提取 | 候选能力清单 | 每条候选都有来源、适用范围和不适用场景。 |
| 写入决策 | 写入计划 | 项目事实不进入通用 AIGC，未验证内容不写成稳定架构知识。 |
| 能力索引更新 | 能力索引或版本记录 | 入口可达，版本影响清楚。 |
| 验证 | 验证记录 | 链路、边界、token 成本和单目标约束已检查。 |

## 判断标准

外部内容进入当前 AIGC 前，必须判断：

- 是否解决重复出现的问题。
- 是否能拓展、补充或完善现有能力。
- 是否能用更少 token 完成功能开发。
- 是否能支持一次只做一件事情。
- 是否已经存在等价能力。
- 是否依赖具体项目事实。
- 是否能被多个项目复用。
- 应写入工作流、规则、模板、能力索引，还是通用架构 wiki。
- 是否有明确验证方式。

## 完成标准

- 本轮只围绕一个能力演化目标。
- 来源和读取边界明确。
- 新增或更新能力能从 `AIGC/capabilities/INDEX.md` 或工作流索引检索到。
- 稳定通用架构知识只进入通用架构 wiki，未验证结论只保留在写入计划或运行记录。
- 版本记录说明修改原因、影响范围和验证方式。
- 没有把外部 harness 的项目事实写入通用 AIGC。
