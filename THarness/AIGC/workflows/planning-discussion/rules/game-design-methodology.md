# 游戏策划方法论规则

本规则把游戏策划讨论从设定描述推进到可开发、可反馈、可验证的体验结构。它不保存书籍原文，也不把书籍摘要当作输出目标。

## 固定沟通流程

1. 捕捉用户原始想法。
2. 追问玩家幻想。
3. 提炼核心体验。
4. 转换成玩法机制。
5. 补充手感、反馈、UI、关卡。
6. 输出可开发的策划结构。
7. 给出最小验证原型。

## 方法卡调用

需要游戏策划判断时，读取 `../method-cards/INDEX.md`，再按以下顺序选择索引：

1. 已知策划阶段时，读取 `../method-cards/phase-index.md`。
2. 只有用户口语表达时，读取 `../method-cards/trigger-index.md`。
3. 命中书籍模块时，读取对应 `../method-cards/cards/*.md`。
4. 需要校准来源时，读取 `../method-cards/source-index.md`。

## 卡片类型

| 卡片 | 用途 |
| --- | --- |
| 原则卡 | 判断设计方向是否成立。 |
| 提问卡 | 在信息不足时追问关键问题。 |
| 转译卡 | 把口语想法转成玩法零件。 |
| 检查卡 | 输出前检查体验、机制、反馈、理解和验证。 |

## 输出要求

游戏策划方案至少区分：

- 原始想法。
- 已确认内容。
- 待确认问题。
- 玩家幻想。
- 核心体验。
- 玩法机制。
- 手感、反馈、UI、关卡承载。
- 最小验证原型。
- 成功标准。

## 禁止项

- 不输出书籍总结。
- 不保存或复述书籍原文。
- 不把“设定”当作“玩法”。
- 不在玩家幻想、核心体验、验证目标不清楚时直接生成完整方案。
- 不替用户确认多解问题。
- 方法卡为 `pending-source` 时，不得声称规则已由书籍原文验证。
