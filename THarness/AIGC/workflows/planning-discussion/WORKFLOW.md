# 策划案讨论工作流

## 目标

把模糊需求讨论成可执行策划案，并把确认后的项目事实同步到目标项目 wiki。

当讨论对象是游戏玩法、战斗体验、系统机制、玩家理解或原型验证时，启用游戏策划方法论模式：问清楚 -> 拆体验 -> 建机制 -> 做反馈 -> 降低理解成本 -> 验证真实需求。

## 激活条件

用户明确表达以下任一意图时激活：

- 启动策划案讨论工作流。
- 讨论一个方案。
- 为某个项目建立策划案。
- 对项目工作流进行前期规划。
- 讨论游戏玩法、战斗体验、系统机制、用户理解或验证原型。

## 默认读取

1. 本文件。
2. `../session-visible-state.md`。
3. `rules/INDEX.md`。
4. `agents/INDEX.md`。
5. 需要执行的 agent 的 `DEFINITION.md` 和 `RULE.md`。
6. `../../wiki/INDEX.md`。
7. 讨论游戏策划问题时读取 `method-cards/INDEX.md`、`rules/game-design-methodology.md` 和命中的方法卡。

## 阶段

| 阶段 | 输出 | 验证 |
| --- | --- | --- |
| 输入澄清 | 未确认点清单 | 关键歧义已显式列出。 |
| 游戏策划方法论 | 玩家幻想、核心体验、机制、反馈和验证原型 | 输出不止是设定，能转成可开发体验结构。 |
| 方案拆解 | 项目 wiki 策划页面草案 | 包含目标、范围、非目标、约束、风险。 |
| 决策确认 | 决策记录 | 用户确认与未确认内容分离。 |
| 项目适配 | 目标项目 wiki 写入计划 | 不向通用 AIGC 写入项目事实。 |
| Wiki 同步 | 架构知识或项目知识 | 通用架构知识进入 `AIGC/wiki/architecture/`，项目知识进入目标项目。 |

## Agent

| Agent | 职责 |
| --- | --- |
| `facilitator` | 主持讨论并提出澄清问题。 |
| `planner` | 输出策划案结构。 |
| `wiki-curator` | 判断知识写入位置与可靠性状态。 |
| `project-adapter` | 为目标项目建立项目适配目录、项目 wiki 与运行记录结构。 |

## 完成标准

- 用户确认策划案或确认继续讨论的阻塞点。
- 项目相关内容没有写入通用 AIGC。
- 已确认的项目策划事实写入或计划写入 `{project_wiki_root}/design/`。
- 未确认的设计问题写入或计划写入 `{project_wiki_root}/open-questions.md`。
- 可复用架构规则只进入通用架构 wiki。
- 目标项目 wiki 有明确写入位置。
