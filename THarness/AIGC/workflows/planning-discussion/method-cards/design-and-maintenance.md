# 游戏策划方法卡设计原理与维护规则

本文说明游戏策划方法卡体系为什么这样设计，以及后续如何导入来源、更新卡片和验证效果。

## 设计原理

游戏策划讨论的核心问题不是“AI 知道多少设定”，而是“AI 能否把模糊想法压成可开发、可反馈、可验证的体验结构”。因此本体系按策划动作组织，而不是按书籍摘要组织。

## 分层原则

| 层级 | 保存内容 | 不保存内容 |
| --- | --- | --- |
| 来源层 | 公开来源链接、用户合法摘录、笔记编号、章节或页码锚点 | 书籍全文、长篇原文、盗版材料 |
| 方法卡层 | 原则卡、提问卡、转译卡、检查卡 | 具体项目玩法事实、数值、剧情、关卡内容 |
| 工作流层 | 何时调用方法卡、如何输出策划结构 | 单本书摘要、未经确认的项目决策 |
| 项目层 | 已确认的项目玩法、运行记录、项目 wiki | 通用 AIGC 方法论 |

这种分层让日常策划讨论只读取方法卡，不反复读取书籍材料；来源材料只用于校准方法卡，不直接变成输出内容。

## 为什么每本书一张方法卡

每本书负责一个策划判断维度：

| 方法卡 | 负责判断 |
| --- | --- |
| `art-of-game-design` | 问清楚玩家幻想和设计问题。 |
| `designing-games` | 把想法转成体验结构。 |
| `game-mechanics` | 把体验转成资源、成长、循环和平衡变量。 |
| `game-feel` | 把机制补成操作、镜头、音效和即时反馈。 |
| `gamers-brain` | 降低理解成本，检查 UX、引导和认知负担。 |
| `sprint` | 把大方案压成最小验证原型。 |
| `mom-test` | 避免诱导式访谈，验证真实行为和需求。 |

这样做的目标是让 AI 在不同策划阶段调用不同判断工具，而不是把所有知识混成一段无法执行的总结。

## 卡片设计原则

- 原则卡回答“这个方向是否成立”。
- 提问卡回答“现在缺什么关键信息”。
- 转译卡回答“用户口语表达如何变成玩法零件”。
- 检查卡回答“输出前哪里可能不成立”。
- 来源锚点回答“这条规则为什么可信”。

方法卡必须直接服务策划产出。如果某段内容不能帮助 AI 追问、转译、检查或验证，就不应该写入方法卡。

## 状态升级规则

| 状态 | 升级条件 |
| --- | --- |
| `pending-source` | 只有卡片槽位，没有合法来源材料。 |
| `partial` | 有公开合法来源或部分合法材料，可形成基础方法，但覆盖有限。 |
| `verified` | 已用合法原文、摘录或笔记校准，并补充章节、页码或笔记编号。 |
| `blocked` | 缺少合法材料，或来源无法确认。 |

从 `partial` 升级到 `verified` 时，必须同时满足：

1. 方法卡内有明确来源锚点。
2. 每条强规则能追溯到来源材料或用户确认的读书笔记。
3. 卡片内容没有复制大段原文。
4. 用真实策划输入验证过卡片能改善输出。

## 维护更新流程

更新方法卡时按以下顺序执行：

1. 确认材料合法性和来源锚点。
2. 判断材料属于哪张方法卡。
3. 只提炼可执行方法，不写书籍摘要。
4. 更新对应 `cards/*.md`。
5. 如新增或改变来源状态，更新 `source-index.md`。
6. 如改变阶段调用关系，更新 `phase-index.md`。
7. 如新增触发表达，更新 `trigger-index.md`。
8. 如影响工作流行为，更新 `game-design-methodology.md`、能力索引和版本记录。
9. 运行 OneHarness 自检。

## 变更边界

- 只补来源、卡片内容或索引时，不改变策划工作流阶段。
- 改变固定沟通流程时，必须更新 `game-design-methodology.md` 并记录能力版本。
- 新增书籍或新方法维度时，必须新增卡片文件、来源索引、阶段或触发索引。
- 删除卡片前必须确认没有阶段索引、触发索引或测试仍引用它。
- 具体项目事实必须写入目标项目适配层，不能写入方法卡目录。

## 验证方式

维护完成后至少执行：

```powershell
python tools\oneharness.py doctor
python tools\oneharness.py gate
python -m unittest tools.test_oneharness
```

如果本次只更新来源内容，还要人工检查：

- 是否没有书籍全文或长摘录进入 `AIGC/`。
- 是否所有强规则都有来源锚点。
- 是否卡片能让 AI 输出玩家幻想、核心体验、机制、反馈、理解成本和最小验证原型。
