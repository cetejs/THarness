# 阶段门控规则

## 目标

防止游戏项目在信息不完整时提前进入实现，或在验证不足时继续扩大内容规模。

## 门控表

| 进入阶段 | 必须具备 | 不满足时 |
| --- | --- | --- |
| 策划到界面合同 | 最小可玩循环、玩家目标、失败路径 | 回到策划讨论。 |
| 界面合同到视觉准备 | 界面清单、按钮、跳转、数据来源 | 补界面拆分。 |
| 视觉准备到素材落地 | 风格基准、占位策略、可用素材格式 | 先做效果图或素材实验。 |
| 素材到数据蓝图 | 素材引用方式、命名规则、缺失策略 | 补素材管线规则。 |
| 数据蓝图到开发前硬门禁 | 工具链、目标语言、默认分辨率、缩放策略、界面合同、素材合同、验证命令 | 补齐开发前检查清单，不进入实现。 |
| 数据蓝图到开发 | 主任务、子任务、接口契约、验证计划 | 不进入代码实现。 |
| 开发到内容扩展 | 垂直切片可玩、核心验证通过 | 先修闭环问题。 |
| 修复到交付 | 复现、修复、回归验证、风险记录 | 继续排查。 |
| 交付到项目侧同步 | 项目入口、状态、验证命令、任务状态、运行记录和追溯索引一致 | 补同步检查，不把交付视为闭环。 |
| 项目经验到通用流程 | 经验不依赖具体项目事实 | 写项目复盘，不写通用流程。 |

## 成功标准写法

成功标准必须写成能独立检查的句子：

- 可运行哪个入口。
- 能完成哪条玩家路径。
- 哪些验证命令必须通过。
- 哪些截图或日志必须能证明结果。
- 哪些行为明确不属于本轮范围。

## 风险处理

- 工具链不确定时，先做环境验证。
- 运行语言不确定时，先验证引擎版本是否支持。
- UI 结果不确定时，先做效果图和截图校准。
- 玩法规模过大时，先收敛成最小闭环。
- 同一问题重复出现时，先加日志和复现入口。

## 开发前硬门禁清单

进入实现前必须集中核对：

- 引擎版本、运行语言、构建入口和运行入口。
- 目标逻辑分辨率、窗口缩放策略和至少一个验收视口。
- 每个玩家可见界面的界面合同、目标设计稿、允许偏差和关键控件 hitbox。
- 素材管线合同，包括源资源、运行时资源、生成脚本、导入缓存刷新方式和缺失策略。
- 数据 schema 或等价结构、样例数据和引用验证方式。
- 主任务、子任务、接口契约、允许修改范围和禁止范围。
- 构建验证、数据验证、最小闭环 smoke、视觉验证、交互验证和回归验证命令或手动步骤。

清单存在未完成项时，只能进入准备或排查阶段；不能默认进入实现。

## 项目侧执行同步门禁

完成交付或通用工作流升级后，必须回查目标项目适配层：

- 项目适配配置中的当前工作流与实际执行工作流一致，或明确说明本轮只做通用沉淀。
- 状态页中的当前阶段、已完成、当前事实、下一步入口和验证结论互相一致。
- 构建、测试和运行入口已经写入项目 source-map 或等价入口，不能只散落在运行记录中。
- 主任务、子任务和接口契约状态能解释本轮完成和未完成范围。
- 运行记录索引、追溯记录索引、CHG 和 VER 成对可查。
- 未完成项、验收失败项和无法自动验证项写入运行记录、开放问题或下一步入口。

存在同步断层时，只能交付为“流程或准备已更新”；不能宣称目标项目开发闭环完成。
