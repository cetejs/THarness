# 调试和可观测性规则

## 目标

让反复出现的运行时问题通过日志、复现和验证闭环解决，不靠猜测反复改。

## 触发条件

出现以下情况时必须进入本规则：

- 用户第二次反馈同类问题。
- 截图和代码推断不一致。
- 运行窗口、设备、分辨率或编辑器状态可能影响结果。
- 构建通过但运行表现不对。
- 本地无法看到用户真实运行环境。

## 必加观测点

按问题类型增加最小日志：

| 问题 | 观测点 |
| --- | --- |
| UI 裁切或错位 | viewport、window、根节点、父容器、目标控件尺寸、缩放策略。 |
| UI 同帧重建或刷新后尺寸累积 | 清空前后子节点数量、移除方式、下一帧重建时机、父容器尺寸、布局刷新触发点。 |
| 素材异常 | 资源路径、纹理原始尺寸、绘制尺寸、加载失败记录。 |
| 数据异常 | 数据 ID、引用目标、缺失字段、默认值来源。 |
| 状态异常 | 当前界面、当前流程节点、关键状态字段。 |
| 构建异常 | 引擎版本、语言运行时版本、入口命令、退出码。 |

## 修复流程

1. 记录用户反馈和证据。
2. 判断是否已有足够日志。
3. 日志不足时先加日志，不直接猜修复。
4. 用最小复现或 headless 启动确认可加载。
5. 修复最小必要范围。
6. 增加静态检查或回归检查，防止同类问题回退。
7. 更新问题记录和验证记录。

## 日志要求

- 日志前缀稳定，便于用户复制和筛选。
- 日志字段要能定位问题，不输出大量无关状态。
- 截图问题至少输出窗口尺寸、逻辑画布尺寸和关键控件尺寸。
- 日志开关应来自配置。

## UI 节点生命周期检查

- 同一帧清空并重建 UI 时，必须先确认目标引擎的节点销毁语义；延迟销毁的引擎不能只把旧节点排队释放。
- Godot 节点树中，如果同一帧会重建 `Control` UI，清空容器时应先从父节点 `RemoveChild(child)`，再调用 `QueueFree()`，避免旧节点继续参与本帧布局。
- 清屏日志最少包含容器路径、清空前子节点数量、清空后子节点数量、是否同帧重建、目标容器尺寸。
- 无法确认生命周期语义时，优先改为下一帧重建或显式刷新布局，不用扩大容器、滚动框或遮罩掩盖布局累积。

## 禁止

- 禁止连续两轮按猜测修改同一问题。
- 禁止只说“本地正常”而不记录用户环境差异。
- 禁止把 headless 验证当作真实窗口截图验证。
- 禁止用扩大面板、隐藏内容或全局滚动条掩盖真实布局问题。
