# 验证和复盘规则

## 目标

把每次游戏开发交付变成可复查的结果，并把稳定经验沉淀到正确位置。

## 验证层级

| 层级 | 目标 |
| --- | --- |
| 构建验证 | 证明项目能编译或导入。 |
| 数据验证 | 证明数据引用、素材引用和配置字段闭合。 |
| smoke 验证 | 证明最小玩家路径可从入口走到结算。 |
| 运行时验证 | 证明引擎能启动目标入口。 |
| 视觉验证 | 证明关键截图与目标效果一致或差异已记录。 |
| 交互验证 | 证明关键按钮、卡牌、列表和跳转能通过真实输入触发，且控件坐标在视口内。 |
| 回归验证 | 证明本次修复的问题不会被轻易改回。 |

## 交付前必须记录

- 修改了什么。
- 为什么改。
- 哪些文件属于本轮范围。
- 跑了哪些验证。
- 关键截图对应的实际控件坐标、尺寸或真实输入结果。
- 关键交互的 hitbox 是否在视口内，以及真实输入是否触发目标状态变化。
- 项目侧执行同步检查结果：当前工作流、状态页、构建运行入口、任务状态、运行记录和追溯记录是否一致。
- 哪些验证无法执行以及影响。
- 剩余风险。
- 用户如何运行或复查。

## 项目侧同步检查

交付前必须确认：

- 当前工作流入口是否与本轮实际工作一致。
- 项目状态页是否反映最新完成项、当前事实、下一步入口和验证边界。
- 构建、测试、运行和验收命令是否进入项目可检索入口。
- 任务、子任务和契约状态是否与运行记录一致。
- CHG、VER 和运行记录索引是否能追溯本轮新增内容。

如果本轮只更新通用工作流，不修改目标项目事实，也要在交付或验证记录中说明项目侧未同步的原因和影响。

## 复盘沉淀

每轮结束后判断：

| 内容 | 写入位置 |
| --- | --- |
| 项目事实、路径、任务、决策、验证结果 | 目标项目 wiki 或运行记录。 |
| 临时命令输出、失败尝试、用户日志 | 目标项目运行记录。 |
| 跨项目稳定方法 | 通用 AIGC 工作流或规则。 |
| 未确认判断 | 目标项目开放问题或运行记录。 |
| 已废弃方案 | 决策记录或问题记录。 |

## 通用化标准

一个经验能写入通用流程，必须满足：

- 不依赖具体项目名称和内容。
- 不依赖单一素材、单一数值或单一文件路径。
- 至少能解释一类问题，而不是一次修复。
- 能转化为阶段、门控、检查项、模板或禁止事项。

## 复盘触发项

出现以下情况时，必须补充工作流或项目规则，而不是只修代码：

- 同类 UI 问题在两个以上界面出现。
- 截图看似正确，但真实点击、拖拽或输入失败。
- 源资源已变更，但运行时仍显示旧资源。
- 为修复一个视觉问题导致相邻界面或同一公共组件的布局回退。
- 用户把某个元素理解为占位或未完成，而实现者原本认为它是功能性 UI。

## 禁止

- 禁止把验证通过写成“应该没问题”。
- 禁止把用户截图中的具体内容写入通用流程。
- 禁止把项目事实复制到 THarness。
- 禁止复盘只写总结，不写下一次可执行的检查项。
