# UI、视觉和运行时规则

## 目标

把玩家可见界面从效果图落到运行时，并让截图、布局、素材和交互能持续校准。

## 界面合同

每个玩家可见界面必须记录：

- 界面目标。
- 主要区域布局。
- 数据来源。
- 按钮和交互。
- 成功反馈、失败反馈和禁用态。
- 跳转来源和跳转目标。
- 是否需要滚动、分页、标签或弹窗。

## 视觉落地顺序

1. 先确认风格基准和参考边界。
2. 生成或整理效果图。
3. 确认目标分辨率、缩放策略和主要视口比例。
4. 把效果图拆成布局区域、层级、尺寸、字体、素材、交互状态和可复用素材类型。
5. 确认哪些是占位素材，哪些是可运行素材。
6. 在运行时用真实素材或占位素材接入。
7. 用截图对比目标效果。
8. 把差异写成具体问题，不用“看起来不对”作为修复目标。

## 设计稿到运行时一致性门控

进入 UI 实现前，必须把已确认的效果图转成运行时可执行合同：

- 记录目标设计稿路径和目标运行分辨率。
- 标出主要区域占屏比例、区域之间的留白、对齐方式和层级顺序。
- 记录每类控件的目标尺寸、最小尺寸、最大尺寸和自适应规则。
- 记录装饰层、背景层、交互层和文本层的遮挡关系。
- 记录每个状态的验收截图：默认态、禁用态、选中态、悬停态、战斗中或结算态。
- 记录允许偏差。没有明确允许偏差时，不能把明显构图差异判定为完成。

## 常见偏差来源

- 把效果图只当成风格参考，没有转成布局和层级合同。
- 只实现通用面板和通用卡牌组件，没有给关键界面单独校准构图。
- 修改默认分辨率后，只放大配置，没有重新确认目标稿或比例合同。
- 自动验证只覆盖构建、数据和 smoke，没有覆盖真实窗口截图。
- 截图反馈只修局部症状，没有回到目标设计稿做整体对齐。
- 日志只能证明控件尺寸和父子关系，不能替代“是否像设计稿”的视觉判断。

## 布局规则

- 默认先设计逻辑画布，再定义窗口缩放策略。
- 固定格式元素要有稳定宽高、比例或配置项。
- 玩家操作区不能被日志区、状态区或装饰区挤压。
- 不用全局滚动框掩盖布局错位。
- 需要滚动时，只让内容列表滚动，不让整个游戏画面都带滚动条。
- 需要适配窗口时，明确是整体缩放、重新排版，还是局部自适应。

## 素材规则

- 参考图不是运行时素材。
- 效果图不是切图交付物，除非已经按运行时尺寸和透明度处理。
- 程序化占位素材必须命名、尺寸和引用稳定。
- 正式素材替换不应改变 UI 代码结构。
- 占位素材不能伪装成最终 UI。空框、空槽、无文本徽章、无功能按钮和只靠边框表达的信息块，必须在验收前移除、补齐语义或明确标注为开发态。
- 生成或覆盖图片资源后，必须刷新引擎导入缓存或使用等效的缓存替换加载策略；不能只看源文件时间戳就判定运行时已经使用新资源。
- 卡牌、道具、技能等核心可收集对象不能只靠边框颜色表达身份。至少要明确费用或等级、名称、类型标识、主体视觉区、效果区和标签或稀有度区。

## 截图校准

截图反馈必须落成：

- 期望效果。
- 实际截图。
- 差异区域。
- 可能来源：素材、布局、缩放、字体、数据或状态。
- 修复后验证方式。

重复截图问题必须进入 `debug-observability.md` 的日志排查流程。

截图只能证明画面状态，不能证明交互可用。含按钮、拖拽、卡牌选择或列表操作的界面，还必须记录关键控件坐标、尺寸和至少一次真实输入结果。

如果一次视觉修复影响公共组件，必须抽查所有复用该组件的界面，尤其是主流程前后相邻界面。不能只验证用户截图所在页面。

## UI 完成判定

UI 完成不是能运行、无编译错误或能点击按钮，而是同时满足：

- 构图、比例、层级和主要留白与目标设计稿一致，或差异已被记录并确认接受。
- 关键控件不会被背景、装饰层、日志区或滚动区遮挡。
- 不存在看起来像占位的空控件、空槽、空面板或没有玩家语义的信息块。
- 窗口缩放后仍符合约定的整体缩放或自适应规则。
- 运行时截图、差异记录和验证结论已经写入项目运行记录。
