# 问题路由规则

## 结论

开发中发现的问题先分类为候选，再由主程审核写入位置，避免经验重复丢失或污染通用层。

## 路由表

| 问题类型 | 写入位置 |
| --- | --- |
| 一次性失败、调试过程、临时命令 | 本轮交付物或目标项目运行记录。 |
| 已确认项目事实、项目模块边界、项目构建测试入口 | 目标项目 wiki。 |
| 项目技术决策 | 目标项目 wiki 的 `decisions/`。 |
| 未确认问题、待用户拍板内容 | 目标项目 wiki 的 `open-questions.md` 或本轮交付物。 |
| 跨项目可复用的架构规则 | `AIGC/wiki/architecture/`。 |
| 通用工作流、门控、验证、写入边界变化 | `AIGC/workflows/` 或 `AIGC/projects/` 对应规则。 |
| 能力入口、状态或版本变化 | `AIGC/capabilities/`。 |

## 处理流程

1. 子任务执行者提交问题候选和证据。
2. 主程判断问题是否只属于当前项目。
3. 主程判断问题是否已经验证或由用户确认。
4. 主程判断后续是否会重复检索。
5. 选择一个写入位置，或记录不写入理由。
6. 更新后执行对应健康检查。

## 禁止

- 禁止把未确认推测写成稳定事实。
- 禁止把一次性调试过程写入 wiki。
- 禁止同一经验在多个规则文件重复维护。
- 禁止从旧项目资料直接复制项目事实到通用 AIGC。
