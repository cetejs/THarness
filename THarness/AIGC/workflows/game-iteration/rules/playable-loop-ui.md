# 最小可玩循环和界面拆分规则

## 目标

把游戏想法转成玩家实际能操作的最小闭环，而不是只输出设定文本。

## 必须输出

- 一句话玩法概念。
- 最小可玩循环。
- 玩家能进入的界面清单。
- 每个界面的整体构成元素。
- 每个界面的 UI 布局。
- 每个按钮的功能、可用条件和跳转目标。
- 无效输入和失败状态反馈。
- 从开始到胜利或失败结算的跳转总表。

## 界面拆分检查

每个界面至少回答：

| 问题 | 说明 |
| --- | --- |
| 玩家为什么来到这里？ | 当前界面的目标。 |
| 玩家能看到什么？ | 主要元素、状态、提示和资源。 |
| 玩家能点什么？ | 按钮、卡片、节点、列表项。 |
| 点了以后发生什么？ | 状态变化、反馈、跳转。 |
| 失败或无效输入怎么反馈？ | 置灰、提示、弹窗、音效或日志。 |
| 当前界面读取什么数据？ | 角色、地图、卡牌、敌人、奖励或存档。 |

## 最小循环验收

最小循环必须满足：

1. 玩家能开始一局。
2. 玩家能做出构筑或策略选择。
3. 玩家能进入核心玩法场景。
4. 玩家能得到明确反馈。
5. 玩家能获得成长或奖励。
6. 玩家能继续推进。
7. 玩家能胜利或失败并进入结算。

## 素材策略提示

当用户想“看到界面长什么样”时，可以先输出低保真界面图或线框图。

当用户想“做可用素材”时，必须提醒：

- 预览图不是正式素材。
- 正式素材应拆成独立透明 PNG 或引擎可复用资源。
- UI 文案、数值和状态应由数据驱动，不嵌死在图片里。

## 禁止

- 禁止只描述世界观，不描述玩家操作。
- 禁止只描述系统，不描述界面入口。
- 禁止让界面跳转出现死路。
- 禁止把“完整游戏”第一步扩大成全部内容量。
