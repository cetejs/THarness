# 游戏迭代工作流

## 目标

把一个游戏想法推进到可持续迭代的项目工作流：先确认玩家可玩的最小循环，再拆界面、素材、数据蓝图和开发门控，最后交接给 `development` 工作流执行。

本工作流只保存跨项目通用流程，不保存具体项目的玩法事实、角色、卡牌、数值、素材路径或运行记录。

## 激活条件

用户明确表达以下任一意图时激活：

- 希望建立一个游戏从策划到开发的迭代流程。
- 需要把游戏讨论过程沉淀为后续可复用工作流。
- 需要判断一个游戏是否已经具备开发前准备。
- 需要为游戏项目补齐玩法循环、界面拆分、数据蓝图、任务门控和验证门控。
- 需要把一个游戏想法持续迭代成可开发任务，但尚未要求直接写代码。

## 与其他工作流的关系

| 工作流 | 关系 |
| --- | --- |
| `planning-discussion` | 用于讨论具体游戏策划、玩法体验、参考游戏结构和未确认问题。 |
| `project-wiki-maintenance` | 用于把目标项目事实写入目标项目 wiki。 |
| `development` | 只有在目标、边界、任务页、契约、子任务和验证方式齐备后，才交接执行开发。 |
| `capability-evolution` | 当本工作流本身需要抽象、升级或版本化时使用。 |

## 默认读取

1. 本文件。
2. `rules/INDEX.md`。
3. 命中阶段规则。
4. 需要项目事实时，通过 `../../projects/INDEX.md` 路由到目标项目适配层。
5. 需要进入策划讨论时，读取 `../planning-discussion/WORKFLOW.md`。
6. 需要进入开发执行时，读取 `../development/WORKFLOW.md`。
7. 需要输出阶段交付物时，读取 `templates/INDEX.md` 和命中的模板。

## 阶段

| 阶段 | 输出 | 验证 |
| --- | --- | --- |
| 输入和边界确认 | 游戏迭代输入摘要 | 目标、参考来源、写入边界、是否允许落文档已明确。 |
| 可玩循环和界面拆分 | 最小可玩循环、界面清单、按钮和跳转 | 玩家能从开始到结算，没有死路界面。 |
| 素材策略 | 素材类型清单、占位素材策略、正式素材边界 | 不把参考图误当正式素材，素材能被控件复用。 |
| 数据蓝图 | 数据模块、字段、引用关系、规模、数值基准 | 所有系统都有数据来源和验收标准。 |
| 开发门控 | 主任务、接口契约、子任务、验证计划 | 没有任务页和契约时不得进入实现。 |
| 交接开发 | development 工作流输入包 | 目标项目 wiki 入口、任务、契约、子任务和验证方式齐备。 |
| 迭代复盘 | 下一轮迭代问题和候选知识 | 项目事实和通用知识写入边界正确。 |

## 完成标准

必须满足：

- 具体项目事实没有写入通用 `AIGC/`。
- 游戏最小循环可以用界面跳转描述。
- 每个玩家可见界面都有数据来源、按钮动作和失败反馈。
- 数据蓝图能覆盖角色、玩法对象、敌人、地图、奖励、UI、素材和存档。
- 开发前存在目标项目 wiki 主任务、接口契约、子任务和验证计划。
- 用户明确确认进入开发后，才能交接 `development` 工作流。

## 禁止

- 禁止把一次项目讨论中的具体角色、卡牌、数值、素材路径或项目名写入本通用工作流。
- 禁止用素材预览图替代正式可复用素材策略。
- 禁止没有任务页和接口契约就进入代码实现。
- 禁止把“完整游戏”理解为第一步填满全部内容；应先验证完整结构能力和最小内容链路。
