一、项目概述

OneHarness 是一套**纯文档型**的低 token、目标驱动的通用 AIGC 工作流框架。它不是代码工具，而是一系列层层套叠的 Markdown 规则文件，用于约束 AI 编码 Agent 的工作方式。整个项目包含约 **80+ 个 Markdown 文件**，按 `workflows/`（工作流）、`wiki/`（通用架构知识）、`projects/`（项目适配模板）、`capabilities/`（能力版本管理）四个顶层分区组织。

---

### 二、优点

1. **层次清晰，路由设计完善**
   每个目录都有 `INDEX.md` 作为入口，每个子文件都标注 `read_when` 触发条件。形成了一个完整的"入口→索引→命中页面"的三级路由体系，AI 可以按需逐层读取，不会一次性吞入所有上下文。

2. **`read_when` 懒加载模式**
   所有规则文件都携带显式的触发条件（如"需求模糊时需要讨论方案"），这与传统 README 式的"全部读完"形成鲜明对比，是核心设计亮点。

3. **知识分层边界极其严格**
   通用框架知识 → `AIGC/wiki/architecture/`；项目事实 → 目标项目适配层；临时过程 → 运行记录。三层隔离避免了历史项目资料污染通用框架。

4. **可验证闭环思想贯穿始终**
   每个工作流阶段都定义了"输出→验证→完成标准"三元组。质量门控 `quality-gate.md` 在交付前对输入完整、范围收敛、验证可执行、边界干净、结果一致进行五项检查。

5. **Wiki 页面 Schema 统一**
   所有架构知识页使用 YAML Front Matter（id、title、summary、tags、relates、read_when、source、updated），支持机器检索和交叉引用。

6. **能力管理与版本控制**
   `capabilities/` 目录包含能力注册表、版本号（当前 0.5.0）、变更日志，对 AIGC 框架自身的演进做了版本化管理。

7. **问题路由表**
   `issue-routing.md` 将开发中发现的问题按类型（一次性失败/项目事实/决策/架构规则/能力变更）精确路由到运行记录、项目 wiki、通用 wiki 或能力索引。

8. **生命周期与多执行者管理**
   `lifecycle.md` 规定了长任务收尾、SubAgent 上下文移交、单一文件写入权的规则，考虑了多人协作场景。

---

### 三、缺点

1. **无实际执行机制——纯"宪法"而无"执法者"**
   框架的全部价值依赖 AI Agent 自觉遵守规则。没有 CLI 工具、没有 lint 脚本、没有 GitHub Action、没有任何自动化验证手段。质量门控、健康检查、写入边界检查全部依靠 AI 自我约束。

2. **自身体量与其宣称的"低 token"矛盾**
   即便按 `read_when` 懒加载，一个首次进入的 AI 会话至少需要读取 15-20 个入口/索引文件才能完成路由。这些文件本身虽然每个很短，但层层套叠的索引结构导致了显著的前置读取成本。

3. **骨架丰满，血肉不足——大量文件沦为占位符**
   多个 Agent 定义文件仅 15-20 行，如 `facilitator/DEFINITION.md` 只有职责/输入/输出三段，缺乏可操作的执行指令。`RULE.md` 文件多处为空或仅有占位内容。结构与内容比严重失衡。

4. **Agent 定义与实际 SubAgent 模型脱节**
   Agent 是按阶段划分的抽象角色（facilitator、planner、coordinator、lead-developer、developer），但其 `DEFINITION.md` 和 `RULE.md` 无法直接转换为可执行的 Agent prompt。代码中也没有对应的 SubAgent 启动脚本或配置。

5. **同步成本被低估**
   `wiki-sync.md` 要求在每一轮完成后将产出同步到 wiki/运行记录，但项目中没有自动化同步工具。这意味同步完全依赖 AI 手动操作，在实际长会话中极易遗漏。

6. **工作流规则缺乏统一的 YAML Schema**
   架构知识页有完善的 YAML 头，但工作流规则文件（`rules/*.md`）仍然是纯 Markdown，没有结构化元数据。这使得规则文件无法被机器索引，检索效率低于 wiki 页面。

7. **缺失关键工作流**
   没有测试工作流、没有发布/部署工作流、没有 Git 提交规范工作流、没有代码审查工作流。当前只有 4 个工作流（策划讨论、开发执行、Wiki 维护、能力演化）。

8. **`index.yaml` 文件形同虚设**
   `wiki/index.yaml` 仅定义了 watch 和 exclude 字段，没有任何实质性的自动索引内容，与外部 harness 的 YAML 自描述索引概念相去甚远。

9. **跨文件内容冗余**
   项目隔离规则同时出现在 `projects/rules/project-isolation.md`、`wiki/architecture/knowledge-boundary.md`、`wiki/architecture/project-adapter-layer.md` 等多个文件中，违反自身"不在索引中复制正文"的原则。

10. **无实际运行验证过**
    从 CHANGELOG 看，所有版本都是"从历史资料提炼"，没有当前项目的实际执行记录或运行验证。框架自身从未被自己的开发执行工作流验证过。

---

### 四、综合评价

OneHarness 是一个**设计理念优秀、但实现停留在文档层面的框架**。它的路由设计、懒加载模式、知识分层思想和闭环验证理念都是高质量的——但这些思想更适合作为一个规范文档存在于某个实际工具之中，而非独立成为一个项目。当前版本 0.5.0 约等于一个"0.1.0 的最小可用规范 + 0.4.0 的架构扩展"，本质上是用大量模板文件填充了一个庞大但尚未落地的框架。
